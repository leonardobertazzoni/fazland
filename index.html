H(yper)T(ext) T(ransfer) P(rotocol)

Il protocollo HTTP e' un protocollo che consente il recupero dei dati. Permette la comunicazione tra client e server tramite dei messaggi testuali (e non binari). Un server HTTP generalmente resta in ascolto delle richieste dei client sulla porta 80 usando il protocollo TCP a livello di trasporto. Questi 2 sono divisi tra loro da 2 proxy (uno per il client e l'altro per il server), componenti intermediari usati per navigare tra reti diverse nell'internet (come il gateway), Nell'uso comune il client corrisponde al browser ed il server alla macchina su cui risiede il sito web.

La natura senza stato (stateless) della sessione di navigazione dell'HTTP costringe gli sviluppatori di contenuti web ad utilizzare dei metodi alternativi, i cookie, per conservare lo stato dell'utente, ma cio' che rende migliore questo protocollo rispetto ad altri e' il fatto che le connessioni vengono generalmente chiuse una volta che una particolare richiesta (o una serie di richieste correlate) e' stata soddisfatta, diminuendo cosi' il numero di connessioni attive limitandole a quelle effettivamente necessarie, con aumento quindi di efficienza (minor carico e occupazione) sia sul client che sul server.

Client e server comunicano tra loro tramite messaggi, request per quello mandato dal client e response per quello mandato dal server.

1. Request

E' costituita da una parte di metodo o verbo seguito dalla versione del protocollo separati da /. Si puo' anche inserire una parte di header che costituisce ulteriori informazioni per il server.

Metodi

sono utilizzati per indicare l'azione che si vuole compiere. Ci sono vari metodi come ad esempio: get, head, post, options e trace.

GET --> serve per "prendere" delle informazioni da un server. Vengono fornite di risposta dal server le informazioni riguardanti il collegamento e la pagina html (il body).

HEAD --> molto simile al metodo get tranne per il fatto che non e' riportato il contenuto della pagina html.

POST --> serve per aggiungere dati al server

OPTIONS --> serve per chiedere al server quali metodi supporta


TRACE --> serve per chiedere al server di restituire una diagnostica


Header

Parte di testo non obbligatoria nella quale client e server aggiungono informazioni nei request/response. Consiste in un nome (key) segiuto da : e il suo valore (i vari campi sono separati tra loro dall'a capo). Sono moltissimi e divisi tra loro in base alla funzionalita'.

2. Response

Status Code

Nel messaggio spedito dal server viene detto se il request e' stato correttamente inviato e quindi accettato o se la comunicazione e' andata a buon fine, in ogni caso viene detto lo stato della connessione. Gli status codes sono divisi in 5 gruppi (basti sulle centinaia) e sono: richiesta ricevuta, continua l'elaborazione - informational responses (100); l'azione e' stata ricevuta con successo, compresa ed accettata - successful responses (200); il client deve eseguire ulteriori azioni per soddisfare la richiesta - redirects (300); la richiesta e' sintatticamente scorretta o non puo' essere soddisfatta - client errors (400); il server ha fallito nel soddisfare una richiesta apparentemente valida - servers errors (500).

Alcuni dei piu' comuni:

  • 200 OK --> la richiesta ha avuto successo
  • 400 Bad Request --> il server non capisce la richiesta per via della sintessai errata
  • 404 Not Found --> il server non trova la risorsa richiesta (nel browser l'URL non e' riconosciuto).
  • 500 Internal Server Error --> server incontra una situazione che non sa come gestire
  • 503 Service Unavaible --> il server non e' in grado di soddisfare la richiesta
